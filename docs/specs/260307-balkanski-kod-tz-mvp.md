# Техническое задание
## Проект: Balkanski kod — угадай мелодию

## 1. Назначение документа
Этот документ описывает требования к MVP веб-приложения **Balkanski kod — угадай мелодию**.

Документ предназначен для разработки проекта в режиме AI-assisted / vibe coding в Cursor под контролем разработчика-куратора. Поэтому требования сформулированы максимально конкретно: с явной бизнес-логикой, структурой данных, экранами, состояниями и ограничениями MVP.

---

## 2. Общее описание проекта
Проект представляет собой небольшую браузерную игру, в которой пользователь слушает музыкальные треки и пытается угадать название песни.

Особенность проекта:
- треки размещаются как статические mp3-файлы на том же сервере;
- список треков поставляется вместе с фронтендом через статичный JSON/TS-объект внутри кода;
- пользователь угадывает название трека по прослушиванию;
- можно использовать подсказки;
- правильным считается ответ как на русском языке, так и на языке оригинала;
- прогресс пользователя хранится локально в браузере.

Проект не предполагает серверной логики, базы данных или авторизации в MVP.

---

## 3. Цель MVP
Минимально жизнеспособная версия должна позволять:
- открыть главную страницу проекта;
- увидеть список доступных треков;
- перейти на страницу конкретного трека;
- начать прослушивание;
- вводить неограниченное количество попыток угадывания;
- получать частичную обратную связь по введенному названию;
- использовать подсказки;
- получить явную подсказку в виде сербского названия;
- угадать трек по русскому или оригинальному названию;
- увидеть карточку угаданного трека с полной информацией;
- видеть общий прогресс и статистику;
- сбросить локальный прогресс.

---

## 4. Что не входит в MVP
В первую версию **не входят**:
- воспроизведение только определенного timestamp-фрагмента трека;
- backend;
- авторизация и пользовательские аккаунты;
- синхронизация прогресса между устройствами;
- импорт/экспорт прогресса;
- фильтры и поиск по трекам;
- i18n;
- светлая тема;
- PWA/offline support;
- unit и e2e тесты;
- SEO-оптимизация;
- отдельный режим dev с альтернативным источником данных.

## 7. Данные и источник контента
### 7.1. Общий принцип
Контент поставляется из статичного объекта / JSON-подобного модуля внутри фронтенд-кода.

Mp3-файлы лежат отдельно в публичной директории сервера:
- `/tracks/...`

Метаданные треков хранятся внутри кода и обновляются через билд.

### 7.2. Контракт данных трека
Для каждого трека должен использоваться следующий контракт:

```ts
export type Track = {
  id: string;
  links: {
    local: string;
    suno: string;
  };
  names: {
    safe: string;
    serbian: string;
    russian: string;
    original: string;
  };
  hints: string[];
  difficulty: number; // 1-5
  dates: {
    added: string; // ISO-like string interpreted as Europe/Belgrade business date
  };
};
```

### 7.3. Правила по полям
#### `id`
- алфавитно-цифровой string;
- используется как основной идентификатор трека;
- отдельный `slug` не нужен;
- должен использоваться в URL.

#### `links.local`
- ссылка на mp3 на том же сервере;
- ожидается публичный путь.

#### `links.suno`
- ссылка на страницу трека в SUNO.

#### `names.safe`
- безопасное отображаемое название до разгадки;
- всегда задается заранее;
- не вычисляется динамически;
- используется в карточках списка и в breadcrumb до момента угадывания.

#### `names.serbian`
- переведенное название на сербском языке;
- используется как явная подсказка;
- после угадывания показывается пользователю в карточке результата.

#### `names.russian`
- русское название;
- является одним из валидных ответов для угадывания;
- всегда одна строка.

#### `names.original`
- оригинальное название;
- является одним из валидных ответов для угадывания;
- всегда одна строка.

#### `hints`
- массив строк;
- может быть пустым;
- обычно содержит 1-3 подсказки;
- подсказки открываются по очереди.

#### `difficulty`
- число от 1 до 5;
- задается вручную;
- используется для отображения в UI и статистики карточки.

#### `dates.added`
- задается вручную;
- хранится строкой;
- должен интерпретироваться как дата добавления трека для логики сортировки и вычисления лейбла «новое».

## 8. Правила игры
### 8.1. Общая механика
Пользователь открывает страницу трека, запускает воспроизведение и пытается угадать название.

Победа засчитывается, если пользователь ввел корректный ответ:
- либо на русском языке;
- либо на языке оригинала.

Количество попыток не ограничено.

### 8.2. Нормализация ответа
Перед сравнением введенный ответ и эталонные ответы должны проходить нормализацию.

Нормализация должна включать:
- приведение к одному регистру;
- trim по краям;
- схлопывание повторных пробелов;
- игнорирование большинства знаков препинания;
- толерантность к несущественным визуальным различиям записи.

### 8.3. Правило правильного ответа
Ответ считается правильным, если:
- он точно совпал с `names.russian` после нормализации;
- или точно совпал с `names.original` после нормализации;
- или совпал по набору значимых слов после нормализации (порядок слов не важен);
- или similarity score после нормализации превышает порог **0.9**.

### 8.4. Частичное совпадение
Если пользователь не угадал название полностью, интерфейс должен пытаться показать частично угаданные элементы.

Правило для MVP:
- частичное совпадение рассчитывается по словам;
- если совпало **20% или более** значимых слов названия, нужно визуально подсветить совпавшие части.

Частичное совпадение:
- не считается отдельной победой;
- не создает отдельный статус трека;
- используется только как UI-обратная связь в стиле Wordle-like угадывания.

### 8.5. Подсказки
Пользователь может нажимать кнопку подсказки.

Логика:
1. если есть неоткрытые элементы в `hints`, открывается следующая подсказка;
2. когда массив подсказок закончился, вместо следующей подсказки становится доступна явная подсказка — показать название на сербском;
3. даже после использования подсказок пользователь может продолжать угадывать;
4. если пользователь угадал после подсказок, трек все равно считается угаданным, но в статистике должен быть отражен факт использования подсказок.

### 8.6. Явная подсказка
Явная подсказка показывает `names.serbian`.

После показа явной подсказки:
- пользователь все еще может продолжать вводить ответы;
- угадывание продолжает засчитываться;
- в статистике должно быть видно, что была использована явная подсказка.

### 8.7. Сдаться
Должна существовать возможность «сдаться / показать ответ».

Если пользователь выбирает этот сценарий:
- трек не считается угаданным обычным способом;
- должен фиксироваться отдельный статус `revealed`;
- карточка трека открывается в режиме раскрытого ответа.

### 8.8. Аудио
Для MVP:
- проигрывается обычный mp3-файл;
- без ограничения на конкретный timestamp-фрагмент;
- необходимы кнопки start / pause.

Перемотка может оставаться стандартной браузерной, если не мешает UX и не требует отдельной доработки.

---

## 9. Состояния трека для пользователя
Для логики приложения необходимо поддерживать следующие состояния пользовательского прогресса по треку:
- `not_started` — трек еще не трогали;
- `in_progress` — были попытки угадывания, но ответа еще нет;
- `solved` — трек угадан;
- `revealed` — пользователь сдался / показал ответ.

Отдельный статус `listened` хранить не нужно.

---

## 10. localStorage и пользовательские данные
### 10.1. Общий принцип
Весь пользовательский прогресс хранится в localStorage.

### 10.2. Требуемая структура
Рекомендуемая структура хранения:

```ts
export type TrackProgressStatus = 'not_started' | 'in_progress' | 'solved' | 'revealed';

export type TrackProgress = {
  trackId: string;
  status: TrackProgressStatus;
  attemptsCount: number;
  attemptsHistory: string[]; // max 20 последних попыток
  hintsUsedCount: number;
  revealedSerbianTitle: boolean;
  revealedByGiveUp: boolean;
  solvedAt?: string; // ISO datetime, заполняется при честном угадывании
};

export type AppStatsSnapshot = {
  resetAt?: string;
};

export type StorageSchema = {
  version: number;
  tracks: Record<string, TrackProgress>;
  stats: AppStatsSnapshot;
};
```

### 10.3. Ограничения на историю попыток
Нужно хранить историю пользовательских попыток, но не бесконечно.

Правило:
- хранить максимум **20 последних попыток** на трек;
- при переполнении работает FIFO:
  - первый элемент уходит;
  - новый добавляется в конец;
- счетчик общего количества попыток при этом продолжает расти.

### 10.4. Что не нужно хранить
Для MVP не требуется хранить:
- дату первого открытия;
- отдельный флаг прослушивания.

Допускается хранить `solvedAt` для формирования виджета «последние угаданные треки».

### 10.5. Версионирование схемы
Нужно заложить `version` в структуру localStorage на будущее.

Допущение для MVP:
- основной контракт данных не ломается;
- структура может расширяться;
- миграции должны быть простыми и безопасными.

### 10.6. Сброс прогресса
На главной странице должна быть кнопка сброса прогресса.

Поведение:
- показывает confirm;
- при подтверждении очищает весь игровой прогресс из localStorage;
- после очистки UI должен вернуться к начальному состоянию.

---

## 11. Логика лейбла «новое»
Трек считается новым, если одновременно выполняются условия:
- с даты `dates.added` прошло не более 30 дней;
- пользователь еще не делал попыток угадывания данного трека.

Если по треку уже была хотя бы одна попытка, лейбл `Новое` больше не показывается.

---

## 12. Статистика
### 12.1. Главная статистика
На главной странице нужно показывать:
- всего треков;
- угадано;
- не угадано;
- начато, но не закончено;
- процент угадывания;
- среднее число попыток;
- сколько треков угадано без подсказок;
- сколько треков угадано с явной подсказкой.

### 12.2. Дополнительные блоки
На главной странице должен быть небольшой виджет:
- последние угаданные треки.

Статистика по сложности в MVP не требуется.

---

## 13. Экран 1 — главная страница
### 13.1. Назначение
Домашняя страница должна быть точкой входа в приложение.

### 13.2. Состав экрана
Обязательные элементы:
- логотип / название проекта;
- список карточек доступных треков;
- блок общей статистики;
- виджет последних угаданных;
- кнопка поделиться проектом;
- кнопка сбросить прогресс;
- ссылка / переход на страницу «О проекте».

### 13.3. Карточка трека на главной
Каждая карточка должна показывать:
- `names.safe`;
- статус трека для текущего пользователя;
- лейбл `Новое`, если применимо;
- сложность;
- число попыток пользователя.

### 13.4. Сортировка
Для MVP карточки сортируются:
- по дате добавления;
- более новые сверху.

Фильтры и поиск не нужны.

---

## 14. Экран 2 — страница игры / трека
### 14.1. Верхняя часть
На странице должны быть:
- логотип / название проекта;
- breadcrumb / ссылка назад;
- заголовок.

Правило заголовка:
- если трек не угадан, показывать `names.safe`;
- если трек угадан или раскрыт, можно показывать `names.serbian`.

### 14.2. Состояние «еще не угадано»
Пока трек не разгадан, карточка должна содержать:
- кнопку «Начать угадывать»;
- после старта — аудиоплеер или контролы start / pause;
- поле ввода ответа;
- визуальную обратную связь по совпавшим частям;
- кнопку отправки ответа;
- кнопку подсказки;
- кнопку явной подсказки, когда обычные подсказки закончились;
- кнопку «Сдаться / показать ответ».

### 14.3. Состояние «угадано»
Если трек угадан, карточка должна показывать:
- сербское название;
- русское название;
- оригинальное название;
- ссылку на SUNO;
- уровень сложности;
- список использованных подсказок / факт их использования;
- информацию о прогрессе пользователя;
- кнопку поделиться карточкой.

### 14.4. Состояние «раскрыто»
Если пользователь сдался, карточка показывает ту же информационную часть, что и для угаданного трека, но статус должен явно отличаться от честного угадывания.

## 15. Экран 3 — «О проекте»
В MVP должен существовать простой экран-заглушка «О проекте».

Он должен содержать:
- краткое описание проекта;
- правила игры;
- при необходимости — информацию о том, что это pet project / музыкальная игра.

Сложного контента на этом экране не требуется.

- Web Share API, если доступен;

### 16.2. Страница трека
Кнопка «Поделиться карточкой» должна работать аналогично:
- Web Share API при наличии;
- fallback на copy link.

Шарится ссылка на конкретную страницу трека внутри проекта.

### 17.1. Тема
Приложение сразу разрабатывается только в dark mode.

### 17.2. Адаптивность
Мобильная адаптация обязательна.

Проект должен корректно работать на:
- десктопе;
- планшете;
- мобильных устройствах.

### 17.3. Анимации и эффекты
Для MVP анимации и звуки интерфейса не требуются.

### 17.4. Язык интерфейса
Весь интерфейс — на русском языке.

i18n в MVP не нужен.

## 18. Маршрутизация
Обязателен React Router.

Минимальный набор маршрутов:
- `/` — главная страница;
- `/track/:id` — страница конкретного трека;
- `/about` — страница «О проекте».

Если трек не найден по `id`, должен открываться fallback-сценарий:
- либо страница 404;
- либо понятный экран «трек не найден» с возвратом на главную.

---

## 19. Обработка ошибок и пограничные случаи
### 19.1. Битый mp3 или недоступный файл
Если mp3 не удалось загрузить:
- приложение не должно падать;
- пользователь должен увидеть понятное сообщение об ошибке;
- остальные части страницы должны оставаться рабочими.

### 19.2. Битая ссылка SUNO
Если ссылка SUNO отсутствует или невалидна:
- блок ссылки не должен ломать карточку;
- допустимо скрыть ссылку или показать ее как недоступную.

### 19.3. Пустые подсказки
Если `hints` пустой массив:
- кнопка подсказки может сразу вести к явной подсказке;
- UI не должен ломаться.

### 19.4. Отсутствующий прогресс в localStorage
Если данных в localStorage нет:
- приложение должно стартовать с чистого состояния без ошибок.

### 19.5. Некорректная версия localStorage
Если структура localStorage отличается от ожидаемой:
- нужно безопасно инициализировать новую схему;
- нельзя допускать падение приложения.

---

## 20. Нефункциональные требования
- приложение должно быть простым для сопровождения;
- код должен быть типизирован TypeScript-типами;
- бизнес-логика проверки ответа должна быть отделена от UI-компонентов;
- работа с localStorage должна быть инкапсулирована в отдельный слой / сервис / model;
- проект должен использовать современный браузерный baseline уровня last 5 browsers или сопоставимый современный target;
- структура проекта должна быть пригодна для дальнейшего расширения списка треков без переработки архитектуры.

---

## 21. Требования к реализации в Cursor / AI-assisted режиме
Чтобы снизить количество ошибок vibe coding, реализация должна следовать этим принципам:

### 21.1. Явное разделение ответственности
Нужно разнести по отдельным сущностям:
- типы треков;
- источник данных треков;
- доменную логику проверки ответа;
- доменную логику подсказок;
- доменную логику вычисления статистики;
- работу с localStorage;
- UI-компоненты.

### 21.2. Минимум скрытой магии
Нельзя строить критичную бизнес-логику на неявных допущениях.

Все важные правила должны быть выражены кодом явно:
- как вычисляется `new`;
- как проверяется правильный ответ;
- как считаются частичные совпадения;
- как считается статистика;
- как работает reveal;
- как хранится прогресс.

### 21.3. Предпочтение простым решениям
Для MVP предпочтительны:
- простые pure functions;
- предсказуемые компоненты;
- отсутствие лишних абстракций;
- отсутствие premature optimization.

## 22. Критерии приемки MVP
MVP считается реализованным, если выполняются все условия ниже.

### 22.1. Контент и навигация
- открывается главная страница;
- отображается список треков;
- работает переход на страницу трека;
- работает страница «О проекте».

### 22.2. Игровая механика
- трек можно воспроизвести;
- пользователь может вводить неограниченные попытки;
- корректный ответ на русском засчитывается;
- корректный ответ на оригинальном языке засчитывается;
- частичные совпадения отображаются в интерфейсе;
- подсказки открываются по очереди;
- после подсказок доступно раскрытие сербского названия;
- можно сдаться и увидеть ответ.

### 22.3. Прогресс
- прогресс сохраняется в localStorage;
- после обновления страницы данные не теряются;
- кнопка сброса очищает прогресс;
- статус карточек корректно обновляется.

### 22.4. Статистика
- статистика на главной считается корректно;
- отображается виджет последних угаданных;
- число попыток пользователя видно на карточках.

### 22.5. UI
- интерфейс работает в dark mode;
- интерфейс адаптивен;
- при битом mp3 приложение не падает.

## 24. Краткий итог для разработчика
Нужно реализовать статическое React + TypeScript + MUI приложение-игру с локальным хранением прогресса, где пользователь слушает mp3, угадывает название трека по русскому или оригинальному имени, получает подсказки, видит статистику и может делиться ссылками на игру и на конкретные карточки.

Главный приоритет разработки:
- предсказуемая бизнес-логика;
- чистая типизация;
- понятный UI;
- легкость дальнейшего ручного добавления новых треков через код.
